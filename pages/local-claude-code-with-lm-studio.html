LM Studio でローカル Claude Code

LM Studio v0.4.1 の Anthropic API 対応を使い、gpt-oss:20B のローカルモデルで Claude Code の反復デバッグ作業まで実用的に動かした検証。

概要

前回の Ollama + gpt-oss:20B 構成では新規コード生成はできても修正作業でループ状態に陥り実用に届かなかった。LM Studio v0.4.1 が Anthropic API 互換エンドポイントに対応したことで、同じ gpt-oss:20B でも修正作業まで完走できた。成功の鍵はコンテキストサイズを 64k に絞り全レイヤーを GPU に乗せたこと、そして KV キャッシュをフル精度で保持したこと。RTX 5060 Ti 16GB という現実的な構成での検証。

詳細

セットアップ

  • LM Studio 公式サイトから v0.4.1 以降をインストールし、GUI から gpt-oss:20B をダウンロード。
  • 「Server」タブでモデルを選択し、コンテキストサイズ 64k・GPU Offload 100%（全レイヤーを GPU へ）・エンドポイント 0.0.0.0:1234（LAN 公開時）で Start Server。
  • Claude Code 側は WSL で ANTHROPIC_API_KEY="lmstudio"（ダミーで可）と ANTHROPIC_BASE_URL="http://<LM StudioのIP>:1234/v1" を設定し、claude --model gpt-oss:20B で起動。

Ollama との違い（成功要因）

項目 Ollama LM Studio
コンテキストサイズ モデルに固定（変更に Modelfile 編集+再ビルド） GUI から自由に変更
KV キャッシュ 量子化あり（精度低下の可能性） フル精度で保持可能
GPU オフロード 自動 手動で 100% 設定可能
パラメータ調整 Modelfile 編集が必要 GUI で即座に調整
  • コンテキスト最適化: Ollama は pull したモデルにサイズが固定され、変更には num_ctx を指定した Modelfile 作成と ollama create の再ビルドが必要。LM Studio は GUI から即変更でき、64k に絞ることで VRAM 16GB に余裕ができ反復会話でメモリを圧迫しない。
  • KV キャッシュ: Ollama は KV キャッシュを量子化しているという報告があり、反復修正で過去コンテキストの精度が落ちてループの原因になった可能性。LM Studio は量子化を GUI でオフにでき、フル精度保持で修正作業が正常動作した。
  • 完全な GPU オフロードと Temperature/Top-p/Repeat Penalty などの GUI 調整の容易さも寄与。

実測

HTML+CSS+JS 単一ファイルのテトリス生成と繰り返しデバッグで、Ollama では実質使えなかった修正作業が LM Studio では完走。速度は簡単なチャットで Ollama 20-45 t/s（CPU オフロード発生）に対し LM Studio（64k）は約 100 t/s（全 GPU）。

補足・結論

コーディング性能自体は qwen3-coder:30B のほうが上（一発で動くコードを出しやすく修正回数を減らせる）で、16GB VRAM への最適化が今後の課題。ローカル LLM ではモデル性能だけでなくランタイム選択とチューニング、特に KV キャッシュの扱いが反復会話で決定的な差を生む。VRAM 16GB では Ollama から LM Studio への切り替えを強く推奨。
