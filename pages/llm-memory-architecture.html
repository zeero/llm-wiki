LLM Memory Architecture

Concepts

LLM Memory Architecture

LLM チャットアシスタントがユーザーの情報を「記憶」しているように見せるためのコンテキスト設計。ChatGPT の実測例では、vector database や会話履歴への RAG を使わず、性質の異なる4層を毎回のプロンプトに合成することで実現している。

概要

LLM 自体はステートレスであり、過去の会話を本質的に「覚えて」はいない。ユーザーが感じる継続性・パーソナライズは、毎メッセージ送信時にモデルへ渡すコンテキストを層状に組み立てることで作られる illusion（錯覚）である。

出典記事は、ChatGPT の振る舞いを会話を通じてリバースエンジニアリングした結果をまとめたもの（OpenAI 非公式・推測ベース）。筆者が「自分について何を覚えているか」を尋ねたところ 33 個の事実が列挙されたが、その裏側は予想よりずっとシンプルで、vector database も会話履歴への RAG も使っていなかった、という発見が主題。

詳細

コンテキスト構造の全体像

ChatGPT が毎メッセージで受け取るコンテキストは以下の順で構成される。

[0] System Instructions        # 高レベルの振る舞い・安全ルール
[1] Developer Instructions
[2] Session Metadata (ephemeral)   # セッション毎に一度だけ注入、永続しない
[3] User Memory (long-term facts)  # 長期の明示的事実
[4] Recent Conversations Summary   # 過去チャットの軽量サマリー（タイトル+抜粋）
[5] Current Session Messages       # 現在の会話（全文・要約なし）
[6] Your latest message

[0] [1] は振る舞いと安全性の定義で本題ではない。記憶機構として重要なのは [2]〜[5] の4層。

第1層: Session Metadata（一時的・非永続）

セッション開始時に一度だけ注入され、長期メモリには入らない。デバイス種別、ブラウザ / user-agent、おおよその location・timezone、subscription level、利用頻度、直近の model 利用分布、画面サイズ、dark mode の有無、JS 有効可否などを含む。会話の depth 平均やユーザーメッセージ長の平均といった利用パターンも含まれる。これはレスポンスを環境に合わせるためのもので、セッション終了後は残らない。

第2層: User Memory（長期の明示的事実）

安定した長期事実を保存・削除する専用の tool を ChatGPT は持つ。数週間〜数ヶ月かけて蓄積し、永続的な「プロフィール」を形成する。名前・年齢、キャリア目標、経歴、進行中プロジェクト、学習中の領域、好みなど。

保存されるのは推測ではなく、次のいずれかの明示的トリガー時のみ。

  • ユーザーが「remember this」「store this in memory」等と言う
  • モデルが OpenAI の基準に合う事実（名前・職種・明言された好み等）を検出し、会話の中でユーザーが暗黙的に同意する

保存された各事実は、以降のすべてのプロンプトに独立したブロックとして注入される。「Store this in memory…」「Delete this from memory…」で追加・削除できる。

第3層: Recent Conversations Summary（軽量サマリー）

過去チャットに対して RAG をかけるのではなく、軽量なダイジェストを使う点が筆者にとって最大の驚きだった。<Timestamp>: <Chat Title> にユーザーメッセージの抜粋（snippet）を添えた形式で、直近の会話を要約する。

  • 要約対象はユーザーのメッセージのみ（アシスタントの発話は含まない）
  • 観測時点でおよそ 15 件のサマリーが利用可能
  • 詳細な文脈ではなく、直近の関心事の「ゆるいマップ」として機能する

これにより full transcript を引き込まずにチャット間の継続性を得る。

第4層: Current Session Messages（sliding window）

現在の会話の通常の sliding window。要約されず全文がそのまま渡される。

  • 上限はメッセージ数ではなく token count ベース
  • 上限に達すると現在セッションの古いメッセージから roll off する（ただし memory facts と conversation summaries は残る）
  • 全文が verbatim で渡されるため、セッション内の一貫した推論が可能

なぜ RAG より優れうるのか

伝統的な RAG は、過去の全メッセージの embedding 化、クエリ毎の類似検索、full context の引き込みを要し、latency と token コストが高い。ChatGPT の方式は軽量サマリーを事前計算して直接注入するだけで、詳細な文脈を犠牲にする代わりに速度と効率を得る。パイプライン全体を自社で制御できる場合、複雑な検索より curated（厳選）でシンプルな方式が上回りうる、という実践的エンジニアリングの教訓になっている。

トレードオフ

明確なトレードオフは「詳細な歴史的文脈 vs. 速度・効率」。ChatGPT は前者を犠牲にするが、大半の会話ではそれが正しいバランスであり、重要なもの（好み・目標・直近の関心）を覚えつつ高速・応答性を保つ。
