pxpipe（プロンプト画像化によるトークン圧縮）

システムプロンプトなど静的で巨大なテキストを PNG 画像に変換して LLM に読ませ、入力トークン費用を大きく削減するローカルプロキシ「pxpipe」の仕組みと限界。

概要

pxpipe は Steven Chong 氏が公開したオープンソースのローカルプロキシ。Claude Code の入力コンテキストを画像化してモデルに渡すことで、Fable 5 の検証でエンドツーエンド 59〜70%（約6割）のコスト削減を達成した。テキストは文字数に比例して課金される一方、画像はピクセル面積で固定トークンが決まる、という LLM 料金体系の構造的ギャップを突いた手法。

詳細

仕組み（情報密度に依存しない画像課金）

Anthropic API では画像は縦横のピクセルサイズで固定トークン数が決まり、画像内にどれだけ細かく文字を書き込んでも消費トークンは変わらない。pxpipe は長大なテキストを幅 1928px の画像内に、視覚エンコーダが認識できるギリギリの密度でぎっしり敷き詰め（行間・字間を空けない、背景と文字のコントラストを最大化）、視覚的なピクセルデータとして渡す。約 4万8000文字（通常約 2万5000トークン）が約 2700 画像トークンで処理され、1 トークンあたりの文字数が約 1 → 約 3.1 に跳ね上がる。API の視覚処理エンジンを事実上の高効率テキストデコーダとして流用している。

プロンプトキャッシュとの使い分け

プロンプトキャッシュ（Anthropic 標準提供）は 5 分〜数十分の間に同じコンテキストへ連続アクセスする用途で有効だが、コーディングで数時間空けて断続的に質問する場合や、RAG で検索結果テキストが毎回入れ替わる場合はキャッシュが揮発・不一致になる。pxpipe の画像化は有効期限に依存せず、いつでも固定の低コストで供給できる。数万行のコーディング規約、変更頻度の低い全体設計図、大規模リポジトリ全体、RAG の複数ドキュメントの動的パックなど、静的で巨大な背景知識の提供に適する。

代償「沈黙の幻覚」と切り分け

テキスト→画像は非可逆（lossy）圧縮で、最大の代償は精度低下。モデルが文字を読み違えてもエラーを出さず、もっともらしい誤答を返す危険がある。ID・ハッシュ・パスワードなど正確な文字列では致命傷になりうる。検証では 12 文字の 16 進数文字列の読み取りで Fable 5 は 15 回中 13 回成功、Opus では全滅。意味を持たないランダム文字列の正確な転写は視覚モデルの苦手分野。このため pxpipe は全入力を画像化せず、正確性が必須の直近対話・精密な修正指示・短いテキストはそのまま送り、大意が掴めればよい静的な指示書・API 仕様書に限定して画像化する。

位置づけ

DeepSeek は視覚モデルでテキストを約 10 倍圧縮しつつ高精度を保つ OCR システムを発表しており、pxpipe は将来モデルが標準実装しうる視覚的コンテキスト圧縮を現行の商業 API 仕様上で先取りしたもの。プロンプトキャッシュなど基本策を導入した上で、多少の読み落としを許容できるワークロードにのみ切るべきカード。普及すれば AI 企業が画像処理のトークン単価を見直す可能性もある。
