Claude Code 拡張競合（Behavioral Drift）

skills / hooks / CLAUDE.md など複数の拡張が同居したときに静かに起きる競合。エラーにならず「なんか遅い・なんか変」として現れるため見つけにくい。

概要

Claude Code の拡張（skills、slash commands、CLAUDE.md、hooks、自動化フレームワーク）は、MCP のような「ツール接続の規約」と異なり、プロンプトレベル・運用レベルで Claude の振る舞いそのものに介入する。合成ルールが標準化されていないため、複数導入時の優先順位や上書き範囲が不定になり、衝突は「エラー」ではなく「挙動のドリフト（Behavioral Drift）」として現れる。Claude は矛盾する指示を例外にせず「それっぽく」整合して続行するため、テストも通ってしまい、原因究明が難しい。

詳細

3つの競合レイヤー

競合タイプ 発生頻度 致命度 壊れ方
Policy（プロンプト合成） 高 中 出力が変わる・遠回りになる・遅くなる
State（共有状態） 中 中 再現しない不具合・途中でやり直す
Hooks 低 最大 完全停止・無限ループ
  • Policy 競合: 複数の「必ず/常に/絶対」が system/injected prompt に同居すると、モデルが全部を満たそうとして遠回りし、フォーマットやツール選択が変わる。最頻出。
  • State 競合: 名称が衝突していなくても、同じファイル・進行管理・worktree/branch を複数ツールが別前提で読み書きすると破綻。再現性が最も悪い。
  • Hooks 競合: 頻度は低いが致命度が最大。SessionStart 等で重い処理・ネットワーク待ち・子プロセスを実行でき、CLI の再帰呼び出しで無限ループ、stdout 汚染でオーケストレーターのパースが指数的に壊れる。原因特定が最も困難。

なぜ見つけにくいか

壊れ方がソフトで例外にならない。LLM が矛盾を自己整合し「慎重になった/賢くなった」ように擬態する。hooks の非決定性 × LLM の非決定性で再現せず、ログを入れると直る（観測効果）。原因はスタックトレースや return code に可視化されない。

対策（分離・観測・ガードレール）

  • Hooks は原則オフをデフォルトに。特に自動化ツール実行時は無効化し、対話時のみ許可。
  • プロファイル分離: 対話用（ワークフロー強制・skills・hooks あり）と自律実行用（最小指示・skills/hooks なし）を用途別に切り替える。
  • Hooks のガードレール（必須）: 再入防止（CLAUDE_HOOK_RUNNING=1 なら即 exit）、タイムアウト（30秒以内）、重い処理禁止、出力は stdout でなく stderr。
  • 有効介入の強制ダンプ: 起動時に有効な hooks 数・CLAUDE.md 行数・skills 数を stderr へ出し、「いつから変わったか」を追える状態にする。
  • 強制語の緩和: CLAUDE.md の must → should、条件付き（複雑なタスクのみ）へ。

診断は Hooks（最優先）→ Policy（強制語 must/always/never のカウント）→ State（設定重複）の順に行うシェルスクリプトが例示されている。
