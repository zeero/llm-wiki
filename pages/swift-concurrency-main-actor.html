Swift Concurrency: MainActor 上かの判定

Concepts

Swift Concurrency: MainActor 上かの判定

Swift 6 の非同期コードで「メインスレッドか」ではなく「どの actor 上か」を問い、MainActor.assertIsolated() などで MainActor 実行を検証する方法。

概要

Swift 6 では、async コンテキストから Thread.isMainThread を参照するとコンパイルエラーになる:

'Thread.isMainThread' is unavailable from asynchronous contexts
Work intended for the main actor should be marked with @MainActor;
this is an error in the Swift 6 language mode.

Swift Concurrency では「どのスレッドか」は実装詳細であり、正しい問いは「どの actor 上か」である。UI 更新は MainActor 上で行う必要があり、重い処理は MainActor の外で走らせる。データ安全性を守るのはスレッドではなく actor isolation であり、actor が直列実行とデータ保護を担う。

詳細

なぜ Thread.isMainThread が使えないか

スレッドは Swift Concurrency ランタイムが管理する実装詳細で、タスクは実行途中でスレッドをまたいで再開しうる。そのため「今どのスレッドか」を問うこと自体が意味を失い、Swift 6 言語モードでは async コンテキストからの Thread.isMainThread 参照をエラーにしている。判定すべきは actor isolation である。

MainActor.assertIsolated() で検証する

開発中に「今 MainActor 上か」を表明する:

func updateUI() {
    MainActor.assertIsolated("UI updates must happen on MainActor!")
    self.title = "Loaded"
}
  • Debug ビルド（Xcode 既定）: MainActor 上でなければ即座にクラッシュする
  • Release ビルド: 何もしない（ゼロコスト）

誤った actor から呼び出された瞬間に開発中に停止するため、問題を早期に発見できる。

全ビルドで停止させたい場合: preconditionIsolated

MainActor.preconditionIsolated("UI must be touched on MainActor")

こちらは Debug / Release 両方でクラッシュする。誤った actor からの呼び出しが本番でロジックエラーとなる場合に使う。

クラッシュ時の見え方（Queue 名が runtime hint）

アサーション失敗時はアプリが trap する。Xcode の debugger navigator で Queue 名を見ると、現在の actor コンテキストが読み取れる:

  • MainActor でない場合: com.apple.root.user-initiated-qos.cooperative (concurrent) のような concurrent queue 名。バックグラウンド actor（例では .userInitiated QoS）上で走っており、UI actor の外にいることを示す
  • MainActor 上の場合: com.apple.main-thread (serial)

Queue 名はあくまで runtime のヒントであり、actor コンテキストを推測する手がかりになる。

カスタム global actor でも同じ手法が使える

自作の actor に対しても assertIsolated / preconditionIsolated が使える:

@globalActor
actor ImageCacheActor {
    static let shared = ImageCacheActor()
}

@ImageCacheActor
func mutateCache() {
    ImageCacheActor.assertIsolated("must be on ImageCacheActor")
    // safe to touch cache here
}

MainActor.assertIsolated と全く同じように機能する。

assertNotMainActor は存在しない

「MainActor 上でないこと」を直接表明する API はない。回避策として一時的に MainActor.assertIsolated("should NOT be main") を差し込み、クラッシュしなければ MainActor 上だと分かる。あるいは breakpoint を張って debug navigator で Task の値を読む。

まとめ

Swift 6 は適切な actor 利用を強制する。Thread.isMainThread を忘れ、「期待した actor 上にいるか？」を問う。開発中は assertIsolated でミスを早期に捕捉し、Xcode の Queue 情報で runtime の actor コンテキストを把握し、カスタムログで actor hop の発生箇所を可視化する。
