Claude Code の権限・設定管理

bypassPermissions に頼らず、階層化された権限制御で Claude Code を安全に運用するためのベストプラクティス。

概要

Claude Code は強力だが、設定の複雑さから多くの開発者が bypassPermissions モードに頼り危険な状態で使いがち。この記事（AI と人間 hikae の対話形式）は、権限制御を「基本的な事故防止」と位置づけ、完璧なセキュリティではなく実用的な多層防御を構築するアプローチを解説する。

詳細

3 つの基本権限モード

  • allowEdits: 基本的な編集を許可（推奨）
  • readOnly: 読み取り専用（学習・調査用）
  • bypassPermissions: 全権限（本当に必要な時のみの「緊急時の最後の手段」）

階層化された権限制御（3 レベル）

設定の優先順位は エンタープライズ > CLI 引数 > ローカル > 共有 > ユーザー設定。

  1. エンタープライズレベル（絶対制御）: /Library/Application Support/ClaudeCode/managed-settings.json。管理者が設置しユーザーは変更不可。Bash(sudo:*)・Read(.env*)・Read(*credential*) などを deny し、disableBypassPermissionsMode: "disable" を設定すれば個人が何をしても --dangerously-skip-permissions は使えない。組織の最低限の基準を保証する「最後の砦」。
  2. プロジェクトレベル（チーム共有）: .claude/settings.json。プロジェクト固有の効率化（Bash(npm run:*) 等の allow）と制約。
  3. 個人レベル（ローカル調整）: .claude/settings.local.json および ~/.claude/settings.json。テーマ・通知・additionalDirectories・環境変数（BASH_DEFAULT_TIMEOUT_MS 等）や個人ツールチェーン（Bash(brew:*), Bash(gh:*) 等）の許可。エンタープライズポリシーに違反しない範囲での個人最適化に徹する。

ネットワーク制御

  • Bash 経由のアクセスが最も危険（.env を base64 化して外部 POST、内部ネットワークスキャン等）。Bash(curl:*)・Bash(wget:*)・Bash(nc:*) を deny。
  • WebFetch/WebSearch は Anthropic サーバー経由のため内部ネットワークへは届かないが、意図しないトークン消費・処理時間増大を防ぐため allow/deny で制御。
  • 非必須トラフィック: CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC: "1" でテレメトリ等を停止。

権限制御の技術的制約

Bash の権限制御は 前方一致のみで、コマンド途中や引数での制御はできない。Bash(curl:*) は効くが、python -c "import requests..."・node -e ...・ssh user@host "curl ..." は防げない。python3 -c・nodejs -e など抜け道も多い。したがって「基本的な事故防止」に留め、本格的なセキュリティはネットワークレベル制御やコンテナ/VM 分離で担う。curl の全面禁止は API 開発・デバッグを不能にするため、最低限のシナリオに絞り柔軟性は許容する。

実装の優先順位・段階的導入

いきなり厳しい設定は開発者が離れるため段階的に導入する（観察期間 allowEdits → 基本制御 deny "Bash(sudo:*)" → エンタープライズポリシー本格運用）。優先順位は (1) 危険コマンドの制御（エンタープライズ）、(2) プロジェクト固有の効率化（チーム）、(3) 個人の利便性、(4) 高度なネットワーク制御（インフラ）。
