デイトレ売買戦略のスコアリング設計

自作デイトレ bot の売買戦略を Claude との対話で把握・改善した記録。スクリーニング → テクニカル計算 → スコアリング → 発注の全体フローと、トレンドゲートが機能不全に陥った原因分析。

概要

Alpaca + n8n で 5 分間隔で動く自作デイトレ bot（Python）。バックテストで有望だった 10 銘柄程度を監視する。売買判定は AND 条件ではなく スコアリング方式（複数のテクニカル指標をポイント合算し閾値超えで発火）を採用し、「強い単一シグナル＋補助」でも発火できる。この記録は Claude にコードを読ませて戦略を把握し、トレードが発生しない問題の原因を突き止めた対話ログ。

全体フロー

5分ごと（n8n）
  ├─ screening.py           出来高急増 or ギャップ or bypass → 通過/除外
  ├─ technical_analysis.py  RSI / MACD / EMA / VWAP / ATR / 15mトレンド 計算
  └─ scoring.py             買い/売りスコア合算 → トレンドゲート適用 → 閾値超えで signal
       ↓
     order.py（ATRベース SL/TP → 発注）

スクリーニング（前段フィルタ）

テクニカル分析を走らせるコストを節約するため「動いている銘柄か」を弾く第一段階。

候補銘柄 → ① NASDAQ100照合 → ② bypass銘柄は無条件通過
→ ③ 流動性フィルタ（avg_volume < 1,000,000 を除外）
→ ④ 価格帯フィルタ（current_price < $5 を除外）
→ ⑤ 出来高急増 OR ギャップ（いずれか1つ満たせば通過）

通過条件（⑤、OR）: 出来高急増（現在出来高 ÷ 平均出来高 ≥ 2.0 倍）／ギャップアップ（(寄り値 − 前日終値) ÷ 前日終値 ≥ +2%）／ギャップダウン（≤ −2%）。bypass volume_surge_ratio gap_threshold min_avg_volume min_price は銘柄別オーバーライド可能。

スコアリング

買いスコア（_calc_buy_score）

指標 条件 スコア
RSI < 30 +3
RSI < 35（排他） +2
MACD ゴールデンクロス +2
EMA ゴールデンクロス +1
出来高比 > 2.0倍 +1
ギャップ率 > 2% +1
VWAP ゴールデンクロス +2
VWAP 近接 かつ RSI < 45 +1
VWAP 上方乖離 > 1.5%（過熱） −1

売りスコア（_calc_sell_score）

RSI > 70 で +3、RSI > 65（排他）で +2、MACD デッドクロス +2、EMA デッドクロス +1、出来高比 > 2.0倍 +1、VWAP デッドクロス +2、VWAP 近接かつ RSI > 55 で +1、VWAP 下方乖離 > 1.5%（過熱）で −1。

シグナル決定ロジック（determine_signal）

  • デフォルト閾値: 買い/売りともに 4 点（銘柄別に config 上書き可能）。
  • トレンドゲート（15分足トレンドで制限）: down → ロング禁止、up → ショート禁止、neutral → 閾値を +1 して慎重判断。
  • 優先順位: buy_score ≥ sell_score なら買いを優先。
  • ATR はスコアリング対象外で、後段の SL/TP 計算用にパススルー。

各指標の設計と注意点

指標 パラメータ 用途
RSI 14期間 過熱・割安判定
MACD 12/26/9 トレンド転換判定
EMA 9/21 短期トレンド判定
VWAP 当日累積 機関投資家の基準値との乖離
ATR 14期間 SL/TP計算用（スコア不使用）
15分足トレンド EMA9/EMA21差 トレンドゲート判定
  • クロス判定: 直前バーと現バーの 2 本比較のみ。クロスした瞬間だけ golden/dead が立ち、次の 5 分足では none に戻る。鮮度は高いがクロス直後の 1 本しか拾えず、n8n のスケジュール精度・API ラグで取りこぼすと消える（5分枠で1分ズレると取りこぼし確率 20% 程度）。改善案は lookback（直近 2〜3 本以内のクロスを有効とみなす）。
  • VWAP: 当日累積のため寄り付き直後はバー数が少なく信頼性が低い。出来高 0 バーは除外。near 判定は ±0.3% 以内（ハードコード）。TSLA のような高ボラ銘柄では狭すぎて VWAP near ボーナスがほぼ死ぬため、per_symbol で near_threshold を調整するのが理想。
  • 15分足トレンドゲート: diff_ratio = abs(EMA9 − EMA21) / EMA21 ≤ neutral_threshold(0.003) なら neutral、EMA9 > EMA21 なら up、EMA9 < EMA21 なら down。

トレンドゲート機能不全の原因分析

trend_15m が約 80% の時間帯で neutral に落ち、neutral 時は閾値が +1 されるため 8 割の時間で閾値が実質 5 点になり、トレードが発生しない主因となっていた。原因は 2 つが重なったもの。

  • 原因①: neutral_threshold が広すぎる（0.3%）: 15 分足の短期 EMA 同士は元々差が開きにくく、この閾値では大半のバーが neutral に落ちる。
  • 原因②: 当日分のバーしか取得していない: n8n の Fetch 15min Bars ノードが new Date().toISOString().slice(0,10)（当日日付のみ）を start に渡すため最大 26 本しか取れない。EMA は本数が少ないと全バーの平均に引っ張られ、EMA9 と EMA21 が似た値に収束する。寄り付き〜午前中は差がほぼ出ず neutral になる。

対応案

  • 対応A（優先度高・工数小）: neutral_threshold を 0.003 → 0.001 に下げる。neutral 率 80% → 30〜40% の見込みだが、原因②が残るため寄り付き直後は改善しない。
  • 対応B（優先度高・工数中）: start を前日（-3日 で週明けの金曜分もカバー、祝日は Alpaca カレンダー API がより安全）に変更し、EMA が育った状態で寄り付きを迎える。午前中の neutral 率が大幅改善。
  • 対応C（優先度低・工数中）: EMA9/21 は両方短期で差が開きにくいため EMA20/50 に変更。時間軸の開きが大きくなり差が出やすい。ただし 50 本以上必要（約 2 営業日分）なので対応 B とセットで実施し、ガード条件を < 50 に変える。

推奨順序は A → B（まず同時適用して数日ログを取り neutral 率が 20〜30% に収まるか確認）→ 安定後に C を検討。

検証が進まないときの案（トレードが発生しない場合）

精度を厳しめにするとトレードが発生せず検証が進まない。案として: A) ペーパートレードモード（閾値を下げてシグナルだけ発火・発注せずログ記録）、B) スコアをログに残しヒストグラム化（閾値 4 に対しスコア 3 止まりの件数を可視化）、C) バックテストで閾値を 3〜6 に振り最適値を探す、D) 銘柄を増やす（スクリーニングが OR 条件なので計算コストは大きく増えない。NASDAQ100 全体も可能だが、閾値・VWAP near 閾値が全銘柄共通のためディフェンシブ銘柄でシグナルの質が落ちる。まずボラ高めの 30〜40 銘柄に絞るのが無難）、E) 時間帯フィルタを一時的に外す。まず A・B を同時にやり、ログが溜まったら C で閾値を決める順が無駄がない。
