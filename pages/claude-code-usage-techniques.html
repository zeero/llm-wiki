Claude Code 活用法（Anthropic 開発チーム）

Anthropic が公式ドキュメントで公開した、Claude Code 開発チーム自身の実際の使い方。X 上の解説スレッド（@0x__tom）による要約。

概要

Claude Code 開発チームの実態として「6ヶ月以上 SQL を1行も書いていない」「先月一度も IDE を開いていない」「Slack のバグ報告スレッドをペーストして『fix』だけで完了」といった働き方が紹介される。コードを「書く」仕事から、AI のワークフローを「設計」する仕事へ移行しているという主張。冒頭には Claude Code 作者 Boris による「これはチームの使い方であり、唯一の正解はない」という趣旨のコメントが添えられている。

詳細

3つの核心テクニック

  • Parallel Workflows（並列ワークフロー）: チームが「一番の生産性向上」と断言。3〜5個の git worktree を同時起動しそれぞれで Claude Code を並列実行、待ち時間ゼロ。worktree に名前を付けシェルエイリアス（za, zb, zc）でワンキー切替。ログ確認や BigQuery だけを行う「分析専用」worktree も用意。
  • Adversarial Reviews（AI 同士のレビュー）: Claude を「部下」でなく「厳しいレビュアー」として使う。1つ目の Claude に計画を書かせ、2つ目に「シニアエンジニアとして厳しく質問し、テストに合格するまで PR を出すな」とレビューさせる。中途半端な修正後は「今わかっている全てを踏まえ、これを捨ててエレガントな解決策を実装して」と指示。
  • Agentic Architecture（サブエージェント活用）: リクエスト末尾に「use subagents」を付けて計算リソースを増やす。個別タスクをサブエージェントにオフロードしメインの文脈を汚さない。フックで権限リクエストを上位モデルへルーティングし安全なものだけ自動承認。

CLAUDE.md で「学習する AI」を作る

同じミスを毎回指摘せず「このミスを二度としないように CLAUDE.md を更新して」と言う。Claude は自分のルールを書くのが得意で、ミス率が測定可能なレベルで下がるまで CLAUDE.md を編集し続ける。タスク/プロジェクトごとに notes ディレクトリを作り PR ごとに更新させる。

カスタムスキルを git にコミット

ルールは「1日に2回以上やることは、スキルかコマンドにしろ」。例: /techdebt（セッション終了時に重複コード検出・削除）、/sync（7日分の Slack・GDrive・Asana・GitHub を一括取得）、dbt モデル作成・コードレビュー・dev テスト用のカスタムエージェント。一度作れば全プロジェクトで再利用する「AI 武器庫」の発想。

その他

  • Plan Mode: いきなり書かせず設計図を完璧に。うまくいかない時は押し進めず計画モードに戻る。8割の時間を計画、2割を実装に。
  • ターミナル環境: 推しは Ghostty。音声入力を推奨（「タイピングの3倍速で話せてプロンプトが詳細になる」、macOS は fn キー2回）。
  • 学習モード: /config で「Learning」出力スタイルを有効化すると変更の理由まで説明。「このコードを説明する HTML プレゼンを作って」で良いスライドが出る。
