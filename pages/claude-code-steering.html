Claude Code の挙動制御（7つの方法）

Claude Code で Claude の振る舞いを指示する7つの手段を、コンテキストコスト・compaction 挙動・権威（指示追従の強さ）の観点で使い分けるための整理。Anthropic 公式ブログ “Steering Claude Code” に基づく。

概要

各手段は「いつコンテキストへ読み込まれるか」「長いセッション（compaction）を通じて持続するか」「どれだけの権威を持つか」で異なる。7つの手段は CLAUDE.md、rules、skills、subagents、hooks、output styles、append-system-prompt。

詳細

一覧

手段 読み込みタイミング compaction コンテキストコスト 主な用途
CLAUDE.md（root） セッション開始・全期間 memoize（一度読みキャッシュ、compaction 後に再読） 高（全行が常時トークン消費） build コマンド、ディレクトリ構成、monorepo 構造、規約、チーム規範
CLAUDE.md（subdir） そのサブディレクトリのファイルを読んだ時 そのディレクトリに触れるまで失われる 低 サブディレクトリ固有の規約
Rules セッション開始（user-level）/ path-scoped は該当ファイル操作時 compaction 時に再注入 中（path-scoped でなければ常時） 特定の制約・規約（例: 全 API handler は Zod で入力検証）
Skills name と description は開始時、body は invoke 時 invoke 済み skill を共有バジェットまで再注入、古い順に drop 低 手続き的ワークフロー（deploy / release チェックリスト）
Subagents name・description・tool list は開始時、body は Agent tool 呼び出し時 最終メッセージ（要約+メタデータ）のみ返る 低（呼ぶまでメインは0コスト、独立コンテキストで実行） 並列作業や隔離すべき副作業（deep search、log 分析、依存監査）
Hooks ライフサイクルイベントで発火 compaction を完全にバイパス 低（設定はメイン外、一部出力のみ返る） 決定論的自動化（linter 実行、Slack 通知、コマンドブロック、PreCompact バックアップ）
Output styles セッション開始・system prompt へ注入 compaction されない 高（system prompt を上書き） 大きな役割変更（コード助手→汎用助手）
append-system-prompt セッション開始・CLI フラグ compaction されない・その invocation のみ 中（初回後は prompt cache） トーン・応答長・書式の好み

各手段のポイント

  • CLAUDE.md: 200行以内・オーナーを置きコードのようにレビュー。索引として他ファイルを指す設計にする。共有リポでは肥大化しやすく、規約は path-scoped rules、手続きは skills へ逃がす。monorepo は各チームの subdirectory に CLAUDE.md を置き claudeMdExcludes で不要分を除外。組織全体の必須事項は MDM 配布の managed CLAUDE.md にする。
  • Rules: .claude/rules/ の markdown。paths frontmatter で path-scoped にすると無関係セッションで読まれない。unscoped rule は CLAUDE.md に書くのと機構的に同じ。
  • Skills: .claude/skills/ の SKILL.md（name/description/body）。手続き（deploy、release チェックリスト、review プロセス）に向く。
  • Subagents: .claude/agents/ の markdown。body は親会話に一切入らず、fresh コンテキストで実行し最終メッセージのみ返る。最大5階層ネスト可。中間結果でメイン会話を汚したくない副作業に使う。skill は手順をメインスレッドで見せ・操舵したい場合。
  • Hooks: command / http / mcp_tool / prompt / agent の型がある。前3つは決定論的、後2つ（prompt / agent）は Claude の判断を使う。PreToolUse で exit code 2 によりツール呼び出しを拒否できる。
  • Output styles: system prompt に座るため最も追従が強い。既定を上書きし、software engineering 助手としての既定指示（変更のスコープ、コメント方針、セキュリティ、テスト実行習慣）を落とすので慎重に。まず組み込みの Proactive / Explanatory / Learning を検討。
  • append-system-prompt: 加算的で役割は変えない。指示が増えるほど追従は逓減する。

置き場所の指針

  • 「毎回 X したら必ず Y」→ hook（モデルの選択ではなく自動実行）。
  • 「絶対にするな」→ 指示ではなく hooks / permissions / managed settings（決定論的な guardrail）。
  • 30行の手続き→ skill。API 固有の rule→ paths: で scope。個人の好み→ user-level のローカルファイル。

これらは skill / subagent / hook / output style を束ねて plugin として共有できる。
