Mac ローカルLLM 高速化ツール比較（mlx-lm / vllm-mlx / oMLX / Ollama / LM Studio）

Apple Silicon Mac（MacBook Pro M3 Max 64GB）で、ローカルLLM推論ツールを同一モデル・同一プロンプトで実測比較した記事（zephel01, 2026年5月）の要約。

概要

Qwen3.6-35B-A3B 4bit を用い、mlx-lm / vllm-mlx / oMLX / Ollama / LM Studio を直列実行で3回平均計測。結論は 「速度なら mlx-lm、API用途なら vllm-mlx、機能性なら oMLX、手軽さなら Ollama」。速度順位は mlx-lm > vllm-mlx > oMLX > Ollama。

詳細

各ツールの位置づけ

  • mlx-lm: 純粋速度の王者。実測でチャット163 / 長文130 / コード91 tok/s と全プロンプトで最速。依存が少ないが起動オプションが多い。OpenAI互換サーバー（mlx_lm.server）を提供。
  • vllm-mlx: API用途の本命（コミュニティ実装 waybarrios/vllm-mlx）。mlx-lm をバックエンドに使うため速度差はわずか5〜7%。--continuous-batching で複数同時セッション対応、OpenAI/Anthropic 両API互換で Claude Code / OpenCode に接続可能。なお vLLM 本体に --backend mlx オプションは存在しない。
  • oMLX: マルチモデル同時ロード + tiered KV キャッシュ（RAM/SSD 2層）+ Continuous Batching + macOS メニューバーアプリ + Admin Dashboard を統合した Apple Silicon 専用サーバー。純速度は mlx-lm の65〜75%だが、運用面（1クリックでモデルロード/アンロード、永続SSDキャッシュ、GUI管理）が強み。--model-dir 配下のサブディレクトリを自動検出する設計。
  • Ollama / LM Studio: 手軽さ・統合性が最強。GUI完結派向け。

本記事最大の発見: Ollama の「MLX採用で爆速」は条件付き

Ollama 0.19+ の「MLXバックエンドで93%向上」という宣伝は、OLLAMA_MLX=1 設定 + 対応モデル使用が必要条件。実測では:

  • Qwen3.5-9B: MLXが確実に効く（チャットで+65%等）。
  • Qwen3.6-35B-A3B（大型MoE）: 効果限定的（最大+17%、コード生成では-11%）。
  • 通常タグ（qwen3.6:35b-a3b）は GGUF Q4_K_M で、デフォルトは llama.cpp の Metal バックエンド動作。未対応モデルはエラーなく Metal に silent fallback するため、ユーザーは速度からしか判断できない。公式の93%向上は Qwen3.5 限定の数値。

実測から見える傾向

  • 長文（rag_8k）ほど tok/s は約20%低下。コード生成（code_4k）は約半分に低下（プリフィルがボトルネック）。
  • DFlash（投機デコード）は公式ベンチで Qwen3.5-9B 4.37倍速等だが、Qwen3.6 MoE + chat 系プロンプトでは逆効果になることも。適用条件の確認が必要。
  • モデル配置: mlx-lm と vllm-mlx は HF キャッシュを共有。oMLX は --model-dir 配下にシンボリックリンク。Ollama は独自ストア（別途17〜24GB）。

M シリーズ世代別の MLX 特性

Apple ML Research によれば、M5 の GPU Neural Accelerator が MLX 高速化の決定的要因。TTFT は compute-bound で M5 が3-4倍速、生成速度は memory-bandwidth-bound で約1.2倍。M3 Max はメモリ帯域400 GB/s と広く、大型モデル（生成が帯域律速）では意外と強い。本記事の結論「mlx-lm 最速 / Ollama 限定的」は M3/M4 Max での結果であり、M5 では評価が変わる可能性がある。
