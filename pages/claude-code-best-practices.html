claude-code-best-practice リポジトリ

GitHub でスター 1,500 超を集めた Claude Code の設定パターン集リポジトリ shanraisshan/claude-code-best-practice と、その実践ガイド。

概要

shanraisshan/claude-code-best-practice（2025年10月作成、”practice made claude perfect”）は、アプリのコードベースではなく Claude Code の設定・ワークフローのテンプレート/リファレンス集。CLAUDE.md の書き方、Command → Agent → Skills の 3 層アーキテクチャ、Hooks 通知、settings.json 全オプション、モノレポでの読み込み挙動、RPI ワークフローなどを網羅する。README の「MY EXPERIENCE」に実運用の経験則がまとまっている点が特に参考になる。

詳細

著者の経験則（MY EXPERIENCE）

  • Workflows
    • CLAUDE.md は 150 行を超えない（長いと守られない指示が増える。ただし 100% 保証ではない）。/memory・/rules・constitution.md は「何も保証しない」とし、CLAUDE.md に直接書くのが最も確実という立場。
    • ワークフローには agents より commands を使う。汎用エージェント（general QA, backend engineer）ではなく、skills（progressive disclosure）を持つ feature specific subagents を作る。
    • コンテキスト使用量は最大 50% で手動 /compact。サブタスクは 50% 以内で完了できるサイズに分割。ステータスラインで常に監視。
    • 小さいタスクにはワークフロー不要（vanilla cc is better than any workflows with smaller tasks）。
    • 常にプランモードから始める。タスク完了ごとに頻繁にコミットする。
  • Utilities: IDE ではなく iTerm（クラッシュ対策）、Wispr Flow による音声プロンプト（10x）、claude-code-voice-hooks、ステータスライン、git worktrees による並列開発、dangerously-skip-permissions の代わりにワイルドカード付き /permissions（Bash(npm run *) 等）、/sandbox による権限プロンプト削減、output styles（学習時 Explanatory / コーチング Learning）、/keybindings。
  • Debugging: /doctor、ログを見たいターミナルはバックグラウンドタスクで実行させる、MCP（Claude in Chrome / Playwright / Chrome DevTool）でコンソールログを直接見せる、スクリーンショット提供。

Command → Agent → Skills の 3 層アーキテクチャ

このリポジトリの中核パターン。天気情報取得を題材にした実装例（weather-orchestration）を含む。

層 配置場所 役割
Command .claude/commands/*.md エントリーポイント。/command-name で起動
Agent .claude/agents/*.md ワークフロー実行主体。Skills をプリロード
Skills .claude/skills/*/SKILL.md ドメイン知識・手順書。Agent 起動時にコンテキストへ注入

例: /weather-orchestrator → weather エージェント起動 → weather-fetcher・weather-transformer スキルがプリロード済み → 手順に従い順次処理。Progressive Disclosure（段階的開示）により、必要な情報が必要なタイミングでだけコンテキストに入る。

なお公式では custom slash commands は skills に統合されており（.claude/commands/ も動くが skills 推奨）、skills はサポートファイル・呼び出し制御・subagent 実行などをサポートする。

Hooks による音声通知

各イベント（ツール使用前後・セッション開始終了・コンパクト前など）でシェルスクリプトを実行。リポジトリでは ElevenLabs TTS 音声で通知（git commit 時は専用音声など）。設定は hooks-config.json（チーム共有）と hooks-config.local.json（個人用・gitignore）の 2 層構造。

settings.json / パーミッション

reports/claude-settings.md に全オプションが網羅され、公式ドキュメントの補完資料として使える。パーミッションはワイルドカード構文（Edit(*), Bash(npm run *), Bash(git *), WebFetch(domain:*), mcp__*）で許可、Read(.env)・Read(./secrets/**) を deny に入れて機密漏洩を防ぐ。

モノレポでの読み込み挙動

  • CLAUDE.md: 起動ディレクトリから上位のものはすべて起動時に読み込み（Ancestor Loading）、サブディレクトリのものはそのファイルにアクセスしたとき遅延読み込み（Descendant Loading）。
  • Skills: 上方向探索はしない。ルートの .claude/skills/ は常に読まれ、サブディレクトリのものはアクセス時に自動検出。description は常にコンテキストに入るが本文は呼び出し時に読み込まれる。

RPI（Research → Plan → Implement）ワークフロー

大きな機能開発向け。/rpi:research（実現可能性分析と GO/NO-GO 判定）→ /rpi:plan（pm.md・ux.md・eng.md・PLAN.md を自動生成）→ /rpi:implement（フェーズごとに実装し code-reviewer がレビュー）。「Research で NO-GO なら止める」設計が無駄な実装を防ぐ。個人開発には重すぎるが、チーム開発・大規模機能では有用。

規模別の取り入れ方

  • 個人・小規模: CLAUDE.md 150 行以内、50% での手動 compact、パーミッション設定、小タスクはワークフローなし。
  • 中〜大規模: 3 層構造、機能特化エージェント、Hooks 通知、モノレポ配置設計。
  • チーム: RPI ワークフロー、settings.json / hooks-config.json のチーム共有、.local.json による個人設定分離。
